## 三、脏数据分析

### 3.1 根因

所有未迁移记录（20,478 条）均为**源库本身的数据完整性问题**，分为两类：

#### 第一类：已删除 Application 的孤儿数据（20,460 条）

源库中部分 Application 被删除后，其关联的下游数据未做级联清理，形成孤儿记录。

**影响链路：**

```
Application 被删除（仅删了 applications 表记录）
│
├── config_items     残留（app_id 指向已删除的 app）
│      ├── commits           残留（config_item_id 指向已删除的 config_item）
│      ├── contents          残留（同上）
│      └── released_config_items  残留（同上）
│
├── releases          残留
├── strategies        残留
├── current_published_strategies  残留
├── kvs               残留
├── released_kvs      残留
├── released_groups   残留
├── released_hooks    残留
├── credential_scopes 残留
└── app_template_variables  残留
```

存在两层脏数据：
- **第一层**：直接引用已删除 App 的记录（`app_id` 找不到映射）
- **第二层**：App 删除导致 config_items 也成为孤儿，进而 commits/contents/released_config_items 的 `config_item_id` 也找不到映射

#### 第二类：已删除业务的孤儿数据（18 条）

部分记录的 `biz_id` 在 `applications` 表中不存在任何对应业务，属于业务本身已被删除后的残留。
