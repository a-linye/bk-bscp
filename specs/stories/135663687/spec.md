# 需求规范：进程托管异常记录数据存储

**需求 ID**：短 ID 135663687 / 长 ID 1020451610135663687
**父需求**：【bscp 进程管理】gse 托管信息检查（短 ID 135190552）
**创建日期**：2026-06-30
**状态**：Draft
**输入**：基于 `req.md` 的需求描述与 `questions.md` 中 Q1–Q6 的澄清结论（均为 resolved_by_doc）

## 概述

为父需求"GSE 托管信息检查"提供**数据基础**：定义并实现「托管异常记录」的数据模型与读写存储，使检查侧（写入异常 / 标记恢复）与操作侧（读取当前是否异常）拥有统一、可追溯的数据载体。本子需求**仅聚焦数据层**，不含检查逻辑、操作拦截逻辑与前端展示/告警。

## 用户场景与测试 *(必需)*

### 用户故事 1 — 异常记录写入与历史保留 (优先级: P1)

作为平台运维，我希望将进程托管检查发现的每一次异常以独立记录持久化、保留历史明细，以便后续追溯每次异常的类型、原因与处理建议。

**优先级理由**：这是整个数据载体的核心写入能力，是检查侧落地异常结论的前提，缺此则后续查询/恢复/操作侧判定均无数据来源，构成最小可用切片。

**独立测试**：可单独通过 DAO 写入一条异常记录并查询验证字段完整性、并对同一进程实例连续写入多条后查询历史返回全部记录来独立验证。

**验收场景**：

1. **Given** 检查侧调用写入接口传入一条异常（含异常类型 / 异常描述 / 处理建议 / 定位信息），**When** 写入执行，**Then** 「托管异常记录」表新增一条记录，字段完整且可查询（对应 AC-001）。
2. **Given** 同一进程实例先后写入多条异常记录，**When** 查询该进程实例的异常历史，**Then** 返回全部历史记录（追加式，非覆盖）（对应 AC-002）。

---

### 用户故事 2 — 异常态判定与恢复 (优先级: P1)

作为操作侧/检查侧，我希望以"最近一次记录状态"判断某进程实例当前是否处于托管异常态，并在检查通过时把对应异常记录标记为已恢复，以便支撑操作侧的异常态判断与异常闭环。

**优先级理由**：判定查询与恢复是操作侧据以拦截/放行的直接依据，是异常闭环的关键能力，与写入能力共同构成本数据层的完整价值。

**独立测试**：可单独构造"存在异常记录""恢复后""无记录""最新为已恢复"等样本，调用判定查询与状态更新接口独立验证返回正确。

**验收场景**：

1. **Given** 某进程实例存在状态为"异常"的记录，**When** 调用状态更新接口将其置为"已恢复"，**Then** 该记录状态变为"已恢复"，且"当前是否异常"判定查询返回否（对应 AC-003）。
2. **Given** 某进程实例无异常记录，或其最近一条记录为"已恢复"，**When** 调用"当前是否异常"判定查询，**Then** 返回否（对应 AC-004）。
3. **Given** 同一进程实例存在 exception 记录与较新的 recovered 记录，**When** 调用"当前是否异常"判定，**Then** 以最新记录为准返回否（对应 AC-T02）。

---

### 边界场景

- **无记录**："当前是否异常"判定在该进程实例无任何记录时返回否。
- **最新已恢复**：同一进程实例历史中既有 exception 又有更新的 recovered 记录时，以最新一条为准，判定返回否。
- **租户隔离**：写入经 `set_tenant_id` 回调自动填充 tenant_id；所有查询均带租户隔离，避免跨租户读到他人记录（对应 AC-T01）。
- **大表查询**：历史明细随检查频次线性累积，按业务 / 进程实例的查询必须走索引，避免全表扫描。
- **写库失败**：写入失败记录日志并返回错误，由调用方决定是否阻断（数据层不自行重试或吞错）。

## 需求 *(必需)*

### 功能需求

- **FR-001**：系统必须新增一张独立的「托管异常记录」表及其数据模型，结构沿用仓库三段式（`ID + Attachment(embedded) + Spec(embedded) + Revision(embedded)`）。
- **FR-002**：异常记录的业务字段（Spec）必须包含：异常类型 `error_type`、异常描述 `error_msg`、处理建议 `handling_suggestion`、记录状态 `status`、检查时间 `checked_at`。
- **FR-003**：异常记录的定位字段（Attachment，冗余存储免 join）必须包含：`tenant_id`、`biz_id`、`host_id`、`process_id`、`process_instance_id`；其中 `tenant_id` 由 `set_tenant_id` 回调自动注入。
- **FR-004**：`error_type` 必须为对标 gsekit 的 string 枚举并提供 `Validate()`，取值为：`PARSING_FAILED`（解析失败）/ `AGENT_EXCEPTION`（agent 异常）/ `ILLEGAL_VALUE_KEY`（非法 valuekey）/ `EXPECTATION_MISMATCH`（配置不符，涵盖"已托管无信息""未托管有信息""属性差异"）/ `OTHER`（其他）。
- **FR-005**：`status` 必须为 string 枚举并提供 `Validate()`，取值为：`exception`（异常）/ `recovered`（已恢复）。
- **FR-006**：系统必须提供写入 DAO 能力——以追加方式新增一条异常记录（非覆盖），记录 ID 由 `id_generators` 分配。
- **FR-007**：系统必须提供查询 DAO 能力——按进程实例 / 业务（带租户隔离）查询异常记录列表，返回全部历史明细。
- **FR-008**：系统必须提供"当前是否异常"判定 DAO 能力——按 `process_instance_id`（+ `biz_id` / `tenant_id`）取最新一条记录（按 id 或 created_at 降序取首条），其 `status==exception` 即为异常，否则（无记录或最新为 recovered）为非异常。
- **FR-009**：系统必须提供状态更新（恢复）DAO 能力——将目标异常记录 `status` 置为 `recovered` 并刷新 `updated_at`（reviser），历史明细仍可追溯。
- **FR-010**：建表 migration 必须建立联合索引 `(biz_id, process_instance_id)`（兼顾"按进程实例取最新记录"判定），并保留 `biz_id` 维度查询能力；`tenant_id` 作为多租户隔离前缀纳入查询条件。
- **FR-011**：新增模型必须注册到 `scripts/gen/main.go` 并通过 `make gen` 生成 `internal/dal/gen/`，新增 DAO 必须挂接到 `internal/dal/dao/dao.go` 的 Set 接口。
- **FR-012**：异常记录不得落入敏感个人信息。

### 范围外（本子需求不包含）

- 定时检查与配置比对逻辑（见子需求"进程托管配置定时检查与异常闭环"）。
- 进程操作拦截逻辑（见子需求"异常托管进程操作拦截"）。
- 异常的前端展示 / 告警。
- 数据归档 / 清理策略（留待后续需求）。
- 审计（audit）接入：异常记录由后台巡检自动写入 / 恢复，非用户资源操作，本期不实现 AuditRes 接口（依据 AGENTS.md 不引入不必要抽象，详见 questions.md Q4）。

### 关键实体 *(涉及数据)*

- **托管异常记录（ProcessManagedException，建议表名 `process_managed_exceptions`，最终命名以 plan 阶段为准）**：一条记录对应"某进程实例某次检查的异常结论"，同一进程实例可追加多条历史记录。
  - 业务字段（Spec）：`error_type`、`error_msg`、`handling_suggestion`、`status`、`checked_at`。
  - 定位字段（Attachment）：`tenant_id`、`biz_id`、`host_id`、`process_id`、`process_instance_id`。
  - 修订字段（Revision，嵌入 `table.Revision`）：creator / reviser / created_at / updated_at。
  - 关系：通过 `process_instance_id` 关联进程实例（`process_instance` 表），通过冗余的 `host_id` / `process_id` / `biz_id` / `tenant_id` 支撑免 join 定位与查询。

## 成功标准 *(必需)*

### 可度量结果

- **SC-001**：检查侧写入一条异常后，可在异常记录数据中查询到该记录且全部业务/定位字段完整无缺失。
- **SC-002**：对同一进程实例写入 N 条异常记录后，历史查询返回的记录数恰为 N（无覆盖、无丢失）。
- **SC-003**：对处于异常态的进程实例执行恢复后，"当前是否异常"判定结果由"是"翻转为"否"。
- **SC-004**：在"无记录""最新为已恢复"两类否定场景下，"当前是否异常"判定 100% 返回否。
- **SC-005**：按业务 / 进程实例的查询走索引命中，不发生全表扫描。
- **SC-006**：写入与查询均在租户隔离下生效，不跨租户读到其他租户记录。

## 假设

- 复用仓库现有数据层链路：GORM + gen 代码生成 + DAO 模式，无新引入框架（依据 req.md 技术方案）。
