## 需求背景

### 业务背景

父需求要求对全业务进程的 GSE 托管配置做一致性检查，并将检查发现的异常进程持久化记录、保留历史明细，供追溯与按建议处置；同时进程操作侧需要据此判断进程实例当前是否处于托管异常态。

本子需求是上述能力的**数据基础**：定义并实现「托管异常记录」的数据模型与读写存储，使检查侧（写入/恢复）与操作侧（读取异常态）有统一、可追溯的数据载体。聚焦数据层，不含检查逻辑与操作拦截逻辑。

### 用户故事

作为 **平台运维**
我想要 **将进程托管检查发现的异常以独立记录持久化并保留历史明细**
以便于 **后续追溯每次异常的类型、原因与处理建议，并支撑操作侧的异常态判断**

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | 新增独立「托管异常记录」表及其数据模型 | P0 | 平台后台 | 字段对标 gsekit |
| F-002 | 提供异常记录的写入、查询、状态更新（恢复）DAO 能力 | P0 | 平台后台 | 供检查侧与操作侧复用 |

### 详细功能描述

#### [F-001] 托管异常记录数据模型

- **输入**：无（建表/模型定义）。
- **处理逻辑**：
  1. 新增独立的「托管异常记录」表，字段对标 gsekit 并满足定位与追溯需求：
     - 异常类型 error_type（解析失败 / agent 异常 / 非法 valuekey / 配置不符 / 已托管无信息 / 未托管有信息 等枚举）；
     - 异常描述 error_msg（含具体差异信息）；
     - 处理建议 handling_suggestion；
     - 关联定位信息：租户、业务、主机、进程实例（process_instance）等；
     - 记录状态（异常 / 已恢复）与检查时间、更新时间。
  2. 历史明细保留：以追加方式保存（非覆盖），可按进程实例追溯历史多条记录。
- **输出**：可被检查侧写入、被操作侧读取的持久化表结构。
- **边界条件**：保留历史明细，不做物理覆盖；表结构需兼容大表查询（按业务/进程实例的索引）。
- **异常处理**：建表/迁移失败按平台通用迁移流程处理。

#### [F-002] 异常记录读写 DAO 能力

- **输入**：检查结论（写入/状态更新）、查询条件（按进程实例/业务）。
- **处理逻辑**：提供以下能力：
  1. 写入一条新的异常记录；
  2. 查询某进程实例/业务的异常记录（含"当前是否处于异常态"的判定查询）；
  3. 将异常记录状态更新为"已恢复"（供下一轮检查通过时调用）。
- **输出**：DAO 接口返回结果。
- **边界条件**：以"最近一次记录状态"判断当前是否异常。
- **异常处理**：写库失败记录日志并返回错误，由调用方决定是否阻断。

### 技术方案概述

- **实现方式**：沿用仓库标准数据层链路落地一张独立的「托管异常记录」表与 DAO：
  migration 建表（`cmd/data-service/db-migration/migrations/`）→ `pkg/dal/table/` 定义模型 →
  `scripts/gen/main.go` 注册并 `make gen` 生成 `internal/dal/gen/` → `internal/dal/dao/` 实现 DAO →
  挂到 `internal/dal/dao/dao.go` 的 Set 接口。
- **涉及模块**：`pkg/dal/table`、`internal/dal/gen`、`internal/dal/dao`、`cmd/data-service/db-migration/migrations`、`scripts/gen`。
- **技术选型**：复用现有 GORM + gen 代码生成与 DAO 模式，无新引入框架。

### 架构影响

- **新增组件**：
  - 新增表模型（建议表名 `process_managed_exceptions`、模型 `ProcessManagedException`，最终命名以 plan 阶段为准），结构沿用 `ID + Attachment(embedded) + Spec(embedded) + Revision(embedded)` 三段式。
  - 新增对应 DAO（写入 / 查询 / 状态更新 / 当前是否异常判定）。
- **变更组件**：`internal/dal/dao/dao.go` 的 Set 接口新增子接口；`scripts/gen/main.go` 注册新模型。
- **数据模型变更**：新增表（见下"数据设计"），不改动既有表结构，向后兼容。
- **向后兼容性**：纯新增，无兼容性影响。

### 数据设计

- **核心实体**：托管异常记录。一条记录对应"某进程实例某次检查的异常结论"，同一进程实例可追加多条历史记录。
- **字段构成**（对标 gsekit + 定位/追溯需求）：
  - 业务字段（Spec）：`error_type`、`error_msg`、`handling_suggestion`、`status`、`checked_at`（检查时间）。
  - 定位字段（Attachment，冗余存储免 join）：`tenant_id`、`biz_id`、`host_id`、`process_id`、`process_instance_id`。
  - Revision：creator / reviser / created_at / updated_at（嵌入 `table.Revision`）。
- **定位字段依据（Q1）**：`ProcessInstanceAttachment` 含 tenant_id/biz_id/process_id/cc_process_id 但**不含 host_id**，host_id 在 `ProcessAttachment`；故异常记录冗余存储 host_id 与 process_instance_id 以支撑定位与查询。tenant_id 由 `internal/dal/dao/set_tenant_id.go` 回调自动注入。
- **error_type 枚举（Q2，对标 gsekit）**：`PARSING_FAILED` / `AGENT_EXCEPTION` / `ILLEGAL_VALUE_KEY` / `EXPECTATION_MISMATCH`（涵盖"已托管无信息/未托管有信息/属性差异"）/ `OTHER`；string 枚举 + `Validate()`，风格对齐 `ProcessStatus`/`AgentStatus`。
- **status 枚举与异常态判定（Q3）**：`status ∈ {exception, recovered}`；"当前是否异常"判定 = 按 `process_instance_id`(+ biz_id/tenant_id) 取最新一条记录（id/created_at 降序首条），`status==exception` 即异常，否则（无记录或最新为 recovered）非异常。
- **历史保留与恢复语义（Q6）**：异常写入追加新行（非覆盖）；恢复时更新目标异常记录 `status=recovered` 并刷新 updated_at（reviser），历史明细仍可追溯。
- **数据量级**：随业务进程实例数与检查频次线性增长，需走索引。
- **索引设计（Q5）**：联合索引 (biz_id, process_instance_id)（兼顾"按进程实例取最新"），保留 biz_id 维度查询；tenant_id 作为多租户隔离前缀纳入查询条件。
- **ID 分配**：记录 ID 由 `id_generators` 表分配（`internal/dal/dao/id.go`）。

### 安全与合规

- **权限控制**：数据层不新增鉴权，沿用调用方（检查侧/操作侧）现有鉴权。
- **审计要求（Q4）**：不接入 audit——异常记录由后台巡检自动写入/恢复，非用户资源操作，父需求与本需求均无审计诉求，依据 AGENTS.md 不引入不必要抽象，本期不实现 AuditRes 接口。
- **数据保护**：异常记录不落入敏感个人信息（沿用父需求安全要求）。
