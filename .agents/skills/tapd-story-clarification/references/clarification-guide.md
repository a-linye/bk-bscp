# 需求澄清最佳实践指南

本指南用于指导 Agent 在需求澄清过程中如何有效地与用户交互，确保需求从"模糊"到
"清晰"的转化。明确的，需求澄清严禁涉及技术实现方案和技术细节。

## 核心原则

需求澄清的本质是将"用户想要的"转化为"团队能做的"，需覆盖四大核心问题：
- **为什么做**（Why）：业务价值和用户痛点
- **做什么**（What）：具体功能和行为
- **怎么做**（How）：技术约束和实现限制
- **怎么算完成**（Done）：可验证的验收标准

## 一、结构化提问法（5W1H）

通过六个维度全面挖掘需求细节：

| 维度 | 问题方向 | 示例 |
|------|---------|------|
| **Who** | 谁在用？ | "目标用户是管理员还是普通用户？" |
| **What** | 做什么？ | "需要展示哪些数据字段？" |
| **Why** | 为什么做？ | "解决的核心问题是什么？" |
| **When** | 什么时候触发？ | "在什么操作之后触发这个流程？" |
| **Where** | 在哪里展示/执行？ | "是后台服务还是需要前端界面？" |
| **How** | 怎么实现？有什么约束？ | "接口协议是 HTTP 还是 gRPC？" |

## 二、澄清交互规范

### 问题整理原则

1. **一次性提出**：将所有问题整理成清单一次性提出，避免碎片式逐条追问
2. **分级排列**：
   - 🔴 **阻塞性问题**放前面：缺少这些信息无法开始实现（如外部接口文档、核心业务规则）
   - 🟡 **非阻塞性问题**放后面：可以先假设后确认（如某个异常场景的处理方式）
3. **数量控制**：每轮提问不超过 10 个问题，超过时分批次提出

### 问题格式

```markdown
### 需要您确认的问题

#### 🔴 阻塞性问题（需要先确认才能继续）

1. **[维度-编号]** 问题描述
   > 背景说明：为什么需要这个信息

#### 🟡 建议确认的问题（可以先用默认假设）

2. **[维度-编号]** 问题描述
   > 如果不确定，我将假设：[默认假设内容]
```

### 多轮对话策略

- 用户提供接口文档路径 → 直接读取文档，提取关键信息，不需要用户手动复制粘贴
- 用户提供设计稿链接 → 记录链接，提取可用信息
- 用户说"这个我不确定" → 记录为"待确认"，提供合理假设
- 用户说"先按你理解的来" → 基于最佳实践做合理假设，标注假设依据

## 三、具体举例法

当需求描述中出现模糊词汇时，引导用户举例具体化：

| 模糊词汇 | 引导方式 |
|---------|---------|
| "灵活" | "能举个不灵活的例子吗？比如什么场景下觉得不够灵活？" |
| "体验好" | "好的体验具体表现是什么？比如响应时间 < 2 秒？操作步骤 ≤ 3 步？" |
| "高性能" | "具体的性能指标是什么？比如 QPS、响应时间 P99？" |
| "安全" | "安全方面有哪些具体要求？比如加密方式、权限控制级别？" |

## 四、BDD 验收标准

使用 Given-When-Then 格式编写验收标准：

```
Given [前置条件/上下文]
When [用户操作/系统事件]
Then [预期结果/系统响应]
```

**示例**：
```
Given 用户已登录且具有管理员角色
When 用户点击"批量配置"按钮并选择 10 个服务实例
Then 系统在 5 秒内完成配置，成功率 ≥ 95%，失败项展示具体错误原因
```

## 五、流程可视化

对于涉及多步骤操作的需求，使用流程描述覆盖：
- **主流程**：正常操作路径
- **异常流程**：出错时的处理方式
- **旁路流程**：特殊条件下的分支路径

## 六、需求质量检查清单

澄清完成后，按以下清单自检：

- [ ] 业务价值是否明确？
- [ ] 目标用户和使用场景是否清楚？
- [ ] 每个功能点都有输入/输出说明？
- [ ] 业务规则是否无歧义？
- [ ] 边界条件和异常处理是否覆盖？
- [ ] 外部系统交互的接口信息是否充分？
- [ ] 性能/安全等非功能需求是否量化？
- [ ] 验收标准是否使用 Given-When-Then 格式？
- [ ] 是否明确了"做什么"和"不做什么"？
- [ ] 未解决问题是否已经全部处理完毕？

## 七、常见误区

1. **"澄清是一次性的"** → 澄清应贯穿全流程，开发中发现问题要及时补充
2. **"澄清是产品经理的事"** → 开发、测试也需参与，确保多视角理解
3. **"模糊先做后改"** → 需求阶段修复成本是编码阶段的 1/10，必须先澄清
4. **澄清包含技术实现细节** → 澄清应关注业务逻辑，技术实现细节在编码阶段处理
