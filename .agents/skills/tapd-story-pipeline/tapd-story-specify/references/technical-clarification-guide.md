# 技术澄清指南

本文档系统梳理需求文档编写完成后、进入开发前的「技术澄清」环节的原则、方法和社区最佳实践，为研发团队提供标准化的技术澄清操作指南。

---

## 一、技术澄清的定位

### 1.1 什么是技术澄清

技术澄清（Technical Clarification / Technical Refinement）是指在需求文档（含用户故事和验收标准）编写完成之后、进入开发实现之前，由技术团队从技术视角对需求进行深度审查、分析和补充的过程。

> **需求澄清**关注的是"做什么"——确保需求描述完整、业务逻辑清晰；
> **技术澄清**关注的是"怎么做"和"能不能做"——确保技术可行、方案合理、风险可控。

### 1.2 在研发流程中的位置

```
需求收集 → 需求澄清 → 需求文档（含用户故事）→ 【技术澄清】（本文档关注的环节） → 开发实现     
```

技术澄清处于**需求文档定稿**与**进入开发**之间的关键衔接点，是研发流水线中不可缺少的质量关卡。

### 1.3 是否需要技术澄清？

**结论：必须进行技术澄清。**

理由如下：

| 维度 | 不做技术澄清的风险 | 做了技术澄清的收益 |
|------|-------------------|-------------------|
| 技术可行性 | 开发中途发现方案不可行，返工成本高 | 提前识别技术障碍，选择可行方案 |
| 架构一致性 | 新功能与现有架构冲突，引入技术债 | 确保架构演进方向一致 |
| 工时准确性 | 估算偏差大，导致迭代规划失准 | 技术细节明确后估算更准确 |
| 外部依赖 | 开发中发现外部接口不可用或不符预期 | 提前确认所有外部依赖可用 |
| 团队对齐 | 开发人员各自理解不同，代码风格混乱 | 技术方案共识，实现路径一致 |
| 测试策略 | 测试团队不清楚技术约束，测试覆盖不足 | 测试策略在开发前就明确 |

**例外情况**（可简化或跳过技术澄清）：
- 纯文案修改、配置变更等极简需求
- 团队已有完全相同的实现经验，且需求文档已包含充分技术细节
- 紧急热修复（事后补充技术记录）

## 二、技术澄清的原则

### 2.1 核心原则

#### 原则一：最小充分信息（Minimum Viable Clarification）
- 技术澄清不是写技术设计文档，而是**补齐开发所需的最小信息集**
- 避免过度设计，只澄清到团队可以自信地开始编码的程度
- 澄清粒度应与需求复杂度成正比

#### 原则二：三方视角对齐（Three Amigos Alignment）
- 产品视角：为什么做、价值是什么、优先级如何权衡
- 开发视角：怎么做、约束是什么、工作量多大
- 测试视角：怎么验证、边界在哪里、风险点是什么

#### 原则三：阻塞优先（Blockers First）
- 优先识别和解决阻塞性问题（缺失的接口文档、不确定的技术选型等）
- 非阻塞性问题可以标记"待确认"，在开发中逐步明确
- 区分"必须知道"和"最好知道"

#### 原则四：事实导向（Fact-Driven）
- 技术澄清应基于事实调查，而非主观猜测
- 不确定的技术问题通过 Spike（技术探针）验证
- 所有技术决策应有明确的依据和权衡记录

#### 原则五：可追溯（Traceable Decisions）
- 每个技术决策记录「决策内容 + 决策人 + 日期 + 理由」
- 待解决问题指派负责人和截止日期
- 技术澄清结果直接关联到需求文档

### 2.2 INVEST + T 就绪标准

技术澄清完成后，需求应达到 **INVEST + T** 标准（在经典 INVEST 基础上增加技术维度）：

| 标准 | 含义 | 技术澄清的检查点 |
|------|------|-----------------|
| **I**ndependent | 独立性 | 技术实现上不强依赖其他未完成需求 |
| **N**egotiable | 可协商 | 技术方案有替代选择，不死锁在唯一路径 |
| **V**aluable | 有价值 | 技术实现能交付可验证的用户价值 |
| **E**stimable | 可估算 | 技术细节充分到可以给出合理工时估算 |
| **S**mall | 足够小 | 技术实现可在单个迭代内完成 |
| **T**estable | 可测试 | 验收标准可直接转化为测试用例 |
| **T**echnically Ready | 技术就绪 | 架构方案已确认、外部依赖已到位、技术风险已评估 |

### 2.3 Definition of Ready（就绪定义）

DoR 是团队共同约定的"进入开发前的准入门槛"。技术澄清的目标就是让需求达到 DoR。

技术就绪检查清单

**A. 故事内容完备性**
- [ ] 用户故事格式完整（As a... I want... So that...）
- [ ] 业务背景和价值描述清晰

**B. 验收标准完备性**
- [ ] 验收标准已编写并审核
- [ ] 覆盖正常路径（Happy Path）
- [ ] 覆盖边界情况和错误场景
- [ ] 非功能需求已明确（性能、安全等）

**C. 技术方案就绪**
- [ ] 技术方案已明确
- [ ] 外部依赖已识别，接口可用性已确认
- [ ] 所需 API / 服务 / 数据源已就绪
- [ ] 测试策略已定义
- [ ] 技术风险已评估并有应对方案

**D. 团队理解对齐**
- [ ] 需求已经充分理解
- [ ] PO 在开发期间可随时回答问题
- [ ] 所有阻塞性问题已解决

## 三、技术澄清的维度

### 3.1 技术可行性

- 当前技术栈是否支持需求实现？是否需要引入新技术？
- 是否存在技术限制（如浏览器兼容性、系统版本要求）？
- 性能目标在现有架构下是否可达成？

### 3.2 架构影响

- 新功能是否与现有架构一致？
- 是否需要新增模块、服务或组件？
- 是否涉及数据模型变更（新增表、字段变更、数据迁移）？
- 是否影响已有功能的稳定性？

### 3.3 外部系统交互

- 是否涉及调用外部系统或被外部系统调用？
- 交互方式是什么（HTTP API、RPC、消息队列、SDK、数据库直连等）？
- 接口关键信息：地址、方法、参数、响应结构、错误码
- 认证方式：API Key、OAuth、mTLS 证书
- 是否有沙箱/测试环境？
- 接口文档是否已获取（Swagger/OpenAPI、Proto、Markdown）？

### 3.4 数据设计

- 核心数据实体和关系是否明确？
- 数据量级和增长预期？
- 是否需要数据迁移？迁移方案是什么？
- 数据一致性要求（强一致 vs 最终一致）？
- 敏感数据的加密和脱敏策略？

### 3.5 安全与合规

- 权限控制方案（RBAC/ABAC）？
- 敏感操作的审计要求？
- 数据传输加密要求？
- 是否涉及合规要求（SOC2、ISO27001、GDPR 等）？

### 3.6 可测试性

- 验收标准能否直接转化为自动化测试？
- 是否需要 Mock 外部服务？
- 测试数据如何准备？
- 是否需要特殊的测试环境配置？

### 3.7 部署与运维

- 是否需要新增配置项？
- 是否需要数据库变更脚本？
- 灰度发布策略是什么？
- 回滚方案是什么？
- 监控和告警如何配置？

### 3.8 非功能性约束

- 响应时间要求（P50/P95/P99）？
- 并发量要求？
- 可用性要求（SLA）？
- 资源限制（CPU/内存/存储/带宽）？

## 四、技术澄清的流程

### 4.1 标准流程

```mermaid
graph TD
    A[需求文档定稿] --> B{需求复杂度评估}
    B -->|简单需求| C[快速技术审查]
    B -->|中等需求| D[标准技术澄清]
    B -->|复杂需求| E[深度技术澄清]
    
    C --> F{达到 DoR？}
    D --> F
    E --> F
    
    F -->|是| G[标记为"技术就绪"<br/>进入开发]
    F -->|否| H[澄清了3次？]

    H -->|是| J[提示用户重新评估需求]
    H -->|否| I[分配澄清任务]
    
    I --> |中等需求| D[标准技术澄清]
    I --> |复杂需求| E[深度技术澄清]
```

### 4.2 简单需求：快速技术审查

适用于配置变更、文案修改、简单 Bug 修复等。

**方式**：开发人员独立审查
**产出**：在需求文档中标注"技术审查通过，无技术风险"

### 4.3 中等需求：标准技术澄清

适用于常规功能开发、已有模式可参考的需求。

**方式**：根据需要开发的功能模块，按需加载技术澄清维度进行澄清
**产出**：按需输出技术澄清内容，将信息合并到需求文档中（同类信息需要合并）

### 4.4 复杂需求：深度技术澄清

适用于涉及新技术引入、架构变更、多系统集成的需求。

**方式**：技术澄清全维度进行澄清
**产出**：完整的技术澄清文档

## 五、技术澄清文档模板

技术澄清完成后，将结果作为需求文档的补充章节写入，推荐以下结构：

```markdown
## 技术澄清

### 技术方案概述
- **实现方式**：[简要描述技术方案]
- **涉及模块**：[影响的代码模块/服务]
- **技术选型**：[框架/工具/库的选择及理由]

### 架构影响
- **新增组件**：[如有]
- **变更组件**：[如有]
- **数据模型变更**：[如有]
- **向后兼容性**：[是否兼容/处理方案]

### 外部依赖

| 依赖项 | 类型 | 状态 | 接口文档 | 负责人 |
|--------|------|------|---------|--------|
| [系统名] | [API/RPC/MQ] | [已确认/待确认] | [链接] | [负责人] |

### 技术风险

| 风险 ID | 风险描述 | 影响 | 概率 | 应对措施 |
|---------|---------|------|------|---------|
| TR-001 | [描述] | [高/中/低] | [高/中/低] | [措施] |

### 技术决策记录

| 决策 | 方案 | 理由 | 决策人 | 日期 |
|------|------|------|--------|------|
| [决策内容] | [选择的方案] | [选择理由] | [姓名] | [日期] |

### 非功能性约束
- **性能目标**：[具体指标]
- **安全要求**：[具体要求]
- **部署约束**：[具体约束]

### 测试策略
- **单元测试**：[覆盖范围]
- **集成测试**：[覆盖范围]
- **端到端测试**：[关键场景]

### 补充的验收标准
- [ ] **AC-T01**：Given [技术前置条件] When [操作] Then [技术验收结果]

### 待解决问题

| 问题 ID | 问题描述 | 负责人 | 截止日期 | 状态 |
|---------|---------|--------|---------|------|
| TQ-001 | [描述] | [负责人] | [日期] | [待确认/已解决] |

```
