# tasks.md 模板与规则

**Phase 2 生成 tasks.md 时必须重新读取本文件。禁止凭记忆生成。**

## 任务类型

| 类型 | 适用场景 | 子任务数 |
|------|---------|---------|
| **TDD 任务** | 新功能、Bug 修复、复杂逻辑 | 5 步 |
| **非 TDD 任务** | 配置修改、重命名、文档更新、依赖升级 | 3 步 |

无法判断时，默认按 TDD 任务处理。

## 模板

````markdown
## 1. <任务组名称>

- [ ] 1.1 <新功能/Bug修复类任务描述>  <!-- TDD 任务 -->
  - [ ] 1.1.1 写失败测试：`<测试文件路径>`
  - [ ] 1.1.2 验证测试失败（运行：`<测试命令>`，确认失败原因是缺少功能）
  - [ ] 1.1.3 写最小实现：`<实现文件路径>`
  - [ ] 1.1.4 验证测试通过（运行：`<测试命令>`，确认所有测试通过）
  - [ ] 1.1.5 重构：整理代码、改善命名、消除重复（保持测试通过）

- [ ] 1.2 <配置修改/文档类任务描述>  <!-- 非 TDD 任务 -->
  - [ ] 1.2.1 执行变更：`<变更文件路径>`
  - [ ] 1.2.2 验证无回归（运行：`<测试或构建命令>`，确认输出干净）
  - [ ] 1.2.3 检查：确认变更范围完整，无遗漏文件或引用

- [ ] 1.3 代码审查  <!-- 编号连续跟随最后一个实现任务 -->
  - 前置：调用 superpowers:verification-before-completion 运行全量测试
  - 调用 superpowers:requesting-code-review 审查本任务组变更
  - 占位符：
    - `{PLAN_OR_REQUIREMENTS}` → `openspec/changes/<name>/specs/*.md` + `tasks.md`
    - `{WHAT_WAS_IMPLEMENTED}` → 本任务组变更文件
    - `{BASE_SHA}` → 任务组开始前 commit SHA
    - `{HEAD_SHA}` → 当前 HEAD
  - Critical/Important → 停等用户指令
  - Minor/无问题 → 自动继续

## 2. <另一个任务组>

- [ ] 2.1 <任务描述>  <!-- TDD 任务 -->
  - [ ] 2.1.1 写失败测试：`<测试文件路径>`
  - [ ] 2.1.2 验证测试失败
  - [ ] 2.1.3 写最小实现：`<实现文件路径>`
  - [ ] 2.1.4 验证测试通过
  - [ ] 2.1.5 重构

- [ ] 2.2 代码审查
  - （同 1.3 格式）

## N. 变更文档定稿 (Required)

**作用域**：仅更新 `openspec/changes/<name>/` 下的文档。主 specs 同步由 `/opsx:archive`（Phase 4）负责。

- [ ] N.1 定稿 design.md: 记录技术决策、偏差和实现细节
- [ ] N.2 定稿 tasks.md: **全量标记**所有层级任务（顶层 + 子任务），将已完成的 `[ ]` 标记为 `[x]`
- [ ] N.3 定稿 proposal.md: 更新范围/影响（若有变化）
- [ ] N.4 定稿 specs/*.md: 更新功能需求（若有变化）+ 验证 delta specs 与实际实现一致
  - 检查 `specs/<service>/spec.md` 中每个 MODIFIED/ADDED 章节是否反映最终实现
  - 若实现与设计有偏差，更新 delta spec 以匹配实际代码
- [ ] N.5 最终校验: 确保所有变更文档反映实际实现
````

## 生成规则

### 代码审查任务编号

审查任务编号**连续跟随该组最后一个实现任务**：
- 实现任务 1.1、1.2 → 审查为 1.3
- 实现任务 2.1~2.4 → 审查为 2.5

禁止使用 `N.X` 占位符，必须使用具体编号。

### 变更文档定稿

- **必须是最后一组任务**
- 子任务为平铺检查项，无需嵌套子任务
- 作用域仅限 `openspec/changes/<name>/`，不涉及主 specs
- N.2 执行时必须全量标记**所有层级**的已完成任务：
  - **顶层任务**（如 `1.1`、`2.2`）和**子任务**（如 `1.1.1`、`2.2.3`）均需检查
  - 禁止只标记顶层任务而遗漏子任务

### 通用规则

- TDD 子任务顺序固定：写失败测试 → 验证失败 → 写最小实现 → 验证通过 → 重构
- 非 TDD 子任务顺序固定：执行变更 → 验证无回归 → 检查完整性
- 文件路径必须精确（相对于项目根目录）
- 测试命令必须可直接运行
- YAGNI：只包含本次变更所需的任务
- 禁止在子任务中添加 Commit 步骤（统一提交）

### TODO 列表完整性

Phase 3 生成 TODO 列表时，必须包含 tasks.md 中的**每一个**顶层任务：
- 所有实现任务
- 所有代码审查任务
- 变更文档定稿

禁止遗漏任何任务。
