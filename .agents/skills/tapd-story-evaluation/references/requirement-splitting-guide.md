# 需求拆分原则与方法

本文档定义需求拆分的核心原则和具体方法，供 `tapd-story-breakdown` 子技能在执行
需求拆分时参考。

---

## 一、拆分原则

### 1. MECE 原则（相互独立，完全穷尽）

- **相互独立（Mutually Exclusive）**：子需求之间无功能重叠，同一功能点不会出现
  在多个子需求中
- **完全穷尽（Collectively Exhaustive）**：所有子需求合在一起完整覆盖父需求的
  全部功能范围

### 2. 单一职责原则

- 每个子需求只解决一个明确的业务问题
- 避免将无关的功能打包在一个子需求中
- 一个子需求对应一个清晰的交付成果

### 3. 可独立测试原则

- 每个子需求必须具备完整的验收标准
- 子需求可以独立进行功能验证
- 不依赖其他子需求即可判断是否完成

### 4. 规模控制原则

- 单个子需求不超过 **2 个用户故事**
- 预估工时建议在 **8-24 人时** 范围内（1-3 人天）
- 超过上限需继续拆分，低于下限可考虑合并

---

## 二、拆分方法

### 1. 按功能模块拆分

适用于需求涉及多个功能模块的场景：

- 将复杂功能分解为独立的子系统
- 识别核心功能与辅助功能
- 每个模块拆为一个子需求

**示例**：
> "用户管理系统" → "用户注册登录" + "角色权限管理" + "用户信息管理"

### 2. 按业务流程拆分

适用于涉及多步骤操作流程的场景：

- 按用户操作流程分段
- 识别关键节点和决策点
- 每个流程段拆为一个子需求

**示例**：
> "订单处理流程" → "订单创建" + "订单支付" + "订单发货" + "订单确认"

### 3. 按技术层次拆分

适用于需求涉及多技术层的场景：

- **前端交互层**：页面、组件、用户交互
- **业务逻辑层**：核心业务规则、状态管理
- **数据访问层**：数据库操作、缓存策略
- **外部集成层**：第三方 API、消息队列等

**示例**：
> "报表功能" → "报表前端展示" + "报表数据计算引擎" + "报表数据采集"

### 4. 按用户角色拆分

适用于需求涉及多角色的场景：

- 按不同角色的操作界面和功能拆分
- 每个角色的功能集作为一个子需求

**示例**：
> "审批系统" → "申请人功能" + "审批人功能" + "管理员功能"

---

## 三、依赖关系分析

拆分后必须明确子需求之间的依赖关系：

### 强依赖
- 必须按顺序实现
- 后续子需求依赖前序子需求的接口、数据结构或功能
- 在文档中使用"依赖: [子需求ID]"标注

### 弱依赖
- 可并行开发
- 共享数据库表或配置但功能独立
- 在文档中使用"建议在 [子需求ID] 之后开发"标注

### 可选依赖
- 根据配置或条件决定是否实现
- 在文档中使用"可选依赖: [子需求ID]"标注

---

## 四、拆分质量检查清单

拆分完成后，按以下清单自检：

- [ ] 每个子需求是否只包含 1-2 个用户故事？
- [ ] 子需求之间是否无功能重叠？
- [ ] 所有子需求合在一起是否覆盖父需求全部功能？
- [ ] 每个子需求是否有明确的验收标准？
- [ ] 依赖关系是否清晰标注？
- [ ] 子需求命名是否准确反映其核心内容？
- [ ] 预估工时是否在合理范围内（8-40 人时）？
