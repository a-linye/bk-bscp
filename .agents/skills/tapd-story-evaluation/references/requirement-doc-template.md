# 子需求文档模板

本模板用于需求拆分后生成规范化的子需求 Markdown 文档。Agent 应根据拆分结果填写
各章节，信息充分的详细填写，信息不足的标注"待确认"。

---

## 模板正文

```markdown
# [子需求名称]

## 基本信息

| 字段 | 值 |
|------|-----|
| 需求 ID | [由 TAPD 创建后填入] |
| 需求名称 | [子需求名称] |
| 父需求 | [父需求名称，没有留空] |
| 父需求ID | [父需求短ID: [短ID] / 长ID: [19位长ID]）没有留空 ] |
| 父需求文档 | [父需求文档在项目中相对路径] |
| 优先级 | [High/Middle/Low/Nice To Have] |
| 价值规模 | [RICE 评分，由 tapd-story-score 填入] |
| 预估工时 | [XX 人时，由 tapd-story-score 填入] |
| 处理人 | [处理人] |
| 创建时间 | [YYYY-MM-DD HH:MM:SS] |
| 原始需求文档 | [本文档在项目中相对路径] |

## 依赖关系

> 如无依赖，必须标注"无依赖，可独立开发"
> 如有依赖，必须在表格中填写依赖需求名称、长ID、依赖原因

| 依赖类型 | 依赖需求 | 需求ID | 说明 |
|---------|---------|------|------|
| [强依赖/弱依赖/可选] | [依赖的子需求名称] | 长ID：[长ID] | [依赖原因] |

## 需求背景

### 业务背景

[从父需求继承的业务背景，聚焦到本子需求相关部分：
- 本子需求解决的具体问题或痛点
- 在整体需求中的定位和作用]

### 用户故事

> 针对需求功能，需要详尽列出。覆盖所有功能点。

作为 [角色]
我想要 [功能/目标]
以便于 [业务价值]

> 最多 2 个用户故事

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | [功能描述] | [P0/P1/P2] | [角色] | [必须/应该有/可以有] |

### 详细功能描述

#### [F-001] [功能名称]

- **输入**：[用户操作或系统触发]
- **处理逻辑**：
  1. [步骤 1]
  2. [步骤 2]
- **输出**：[系统响应或结果]
- **边界条件**：
  - [最大值/最小值/特殊情况]
- **异常处理**：
  - [错误场景] → [处理方式]

## 非功能需求

### 性能需求
- **响应时间**：[具体指标]
- **并发能力**：[具体指标]

### 安全需求
- **权限控制**：[具体要求]
- **数据保护**：[加密/脱敏要求]

## 验收标准

### 功能验收

- [ ] **AC-001**：Given [前置条件] When [操作] Then [预期结果]
- [ ] **AC-002**：Given [前置条件] When [操作] Then [预期结果]

### 性能验收

- [ ] **AC-P01**：[性能指标及达标标准]

## 边界范围

### 本子需求包含
- [功能 1]
- [功能 2]

### 本子需求不包含
- [排除项 1]
- [排除项 2]

## 人力与工时

* 全量工作1位高级工程师完成工时预估：[XX 人时]
* 全量工作1位中级工程师完成工时预估：[XX 人时]

## RICE 评分明细

| 参数 | 值 | 说明 |
|------|-----|------|
| Reach | [值] | [说明] |
| Impact | [值] | [说明] |
| Confidence | [百分比] | [说明] |
| Effort | [人天数] | [预估工时 / 8] |
| **RICE Score** | **[得分]** | [优先级建议] |
```

## 模板使用说明

### 拆分阶段填写

由 `tapd-story-breakdown` 在拆分阶段填写以下章节：
- 基本信息（除"价值规模"和"预估工时"外）
- 依赖关系
- 需求背景
- 用户故事
- 功能需求
- 非功能需求
- 验收标准
- 边界范围

### 评分阶段填写

由 `tapd-story-score` 在评分阶段补充以下章节：
- 基本信息中的"价值规模"和"预估工时"
- 人力与工时
- RICE 评分明细

### 文档质量标准

- 避免使用"较快""较好""灵活"等模糊词汇
- 所有性能指标必须使用可量化的数字
- 验收标准必须使用 Given-When-Then 格式
- 用户故事不超过 2 个
- 每个功能点必须有输入/输出说明
