# 评审意见撰写规范

好的评审意见应该帮助开发者成长，而不是打击积极性。

## 核心原则

1. **对事不对人** - 评论代码，而不是评论开发者
2. **解释为什么** - 帮助开发者理解原因，传递知识
3. **提供建议** - 给出具体的改进方案
4. **保持礼貌** - 使用建设性的语言

## 礼貌原则

**示例对比：**

```
❌ 糟糕：
"为什么这里你使用了线程，显然并发并没有带来什么好处？"

✅ 好的：
"这里的并发模型增加了系统的复杂性，但没有任何实际的性能优势。
因为没有性能优势，最好是将这些代码作为单线程处理而不是使用多线程。"
```

## 解释为什么

**示例：**

```
这里建议使用 `const` 代替 `let`，因为 `total` 在后续代码中没有被重新赋值。
使用 `const` 可以让其他开发者明确知道这个变量不会被修改，提高代码可读性。
参考：JavaScript 编码规范 2.1 节
```

## 评论格式模板

```
[必须/建议/Nit] 规则ID: 问题简述

问题: 具体描述问题所在
原因: 解释为什么这是一个问题
建议: 提供具体的修复方案
参考: 相关规范或文档链接（可选）
```
