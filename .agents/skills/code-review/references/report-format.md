# 评审报告格式说明

评审报告采用结构化格式，便于快速理解和处理。

## 报告结构

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
代码评审报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

项目: {项目名}
评审范围: {文件/分支}
评审时间: {时间}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
评审结论
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📋 结论: {LGTM/需要修改/...}

🔴 必须修改: N 个问题
🟡 建议修改: N 个问题
⚪ 可选优化: N 个 Nit

📊 代码质量评分: XX/100

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
问题详情
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔴 {规则ID}: {问题标题}
   文件: {文件路径}:{行号}
   问题: {问题描述}
   建议: {修复建议}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
亮点 & 下一步行动
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 结论类型

| 结论 | 含义 | 行动 |
|------|------|------|
| `LGTM` | Looks Good To Me | 可以合入 |
| `需要小修改` | 有小问题需要处理 | 修复后可合入 |
| `需要修改` | 有明显问题需要修复 | 修复后重新评审 |
| `需要重大修改` | 有设计级别问题 | 需要讨论方案 |

## JSON 报告格式

```json
{
  "summary": {
    "verdict": "需要修改",
    "issuesCount": { "critical": 2, "major": 3, "minor": 5 },
    "score": 75
  },
  "issues": [{
    "ruleId": "FUNC-002",
    "severity": "critical",
    "file": "src/utils/helper.js",
    "line": 42,
    "issue": "未处理边缘情况",
    "suggestion": "添加空值检查"
  }],
  "positives": ["代码结构清晰"],
  "recommendations": ["建议添加单元测试"]
}
```
