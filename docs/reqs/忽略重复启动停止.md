## 需求背景

### 业务背景

进程批量操作（启动/停止等）通过标准运维（SOPS）插件调用，插件在轮询批次结果时会传入
`ignore_error_codes`（需忽略的错误码列表），命中这些错误码的失败子任务在批次整体状态
重判时按成功计入，从而避免"无意义但无害"的失败影响批次成败结论。

当前存在两类"无害失败"未被完整纳入忽略机制：

- **重复启动**：对一个已经在运行的进程再次执行启动。GSE 侧用错误码 `828` 表示
  （进程已在运行，拒绝重复启动）。目前仅在「对比 GSE 状态阶段」和「实际操作阶段」
  记录了 `828`，而最前置的「操作校验阶段」（基于 bscp 库内状态判定"无需启动"）
  直接返回失败、未记录任何错误码，导致这一路径上的重复启动无法被忽略。
- **重复停止**：对一个已经停止的进程再次执行停止。GSE 侧用错误码 `829` 表示
  （`no need to stop the process that is not running`）。目前全流程均未将其纳入
  可忽略错误码，重复停止会被判为真实失败。

不处理会导致：用户对一批进程做批量启动/停止时，其中本就处于目标态的进程会被算作失败，
批次整体被判为"失败/部分失败"，与用户预期（这些进程已是目标态，应视为成功）不符，
造成误报。

### 用户故事

作为 **使用标准运维插件做进程批量操作的运维人员**
我想要 **对已在运行的进程重复启动、对已停止的进程重复停止时，这些子任务不影响批次整体成败结论**
以便于 **批量操作结果只反映真正需要关注的失败，避免"无需操作"被误报为失败**

### 需求来源

- **需求渠道**：技术优化 / 线上误报反馈
- **关联需求**：已有 `ignore_error_codes`（忽略错误码）批次状态重判机制
- **参考资料**：GSE 进程操作接口返回示例（错误码 829，见"原需求描述"）

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | 校验阶段命中"重复启动"时记录错误码 828 | P0 | 进程操作执行器 | 补齐现状缺口 |
| F-002 | 全流程命中"重复停止/强制停止"时记录错误码 829 | P0 | 进程操作执行器 | 新增，stop 与 kill 一致 |

### 详细功能描述

#### [F-001] 校验阶段补齐"重复启动"错误码

- **输入**：对一个 bscp 库内状态已为"运行中"的进程执行启动（start）操作，进入操作校验阶段。
- **处理逻辑**：
  1. 校验判定该启动操作"无需执行"（进程已启动）。
  2. 在返回失败的同时，记录可忽略错误码 `828`（与"对比 GSE 状态阶段""实际操作阶段"
     现有的 828 记录保持一致）。
- **输出**：子任务失败，且携带错误码 `828`，可被查询侧 `ignore_error_codes` 忽略。
- **边界条件**：仅 start 操作命中"已启动/无需启动"时记录 828。
- **异常处理**：其他失败原因（如命令未配置、状态未知、配置不一致等）不记录 828，保持真实失败。

#### [F-002] 全流程支持"重复停止/强制停止"错误码 829

- **输入**：对一个已停止的进程执行停止（stop）或强制停止（kill）操作。kill 的判定逻辑
  与 stop 完全一致（两者在校验阶段、对比 GSE 阶段本就归为同一分支处理）。
- **处理逻辑**：在以下三个阶段命中"进程已停止、无需停止/强制停止"时，记录可忽略错误码 `829`：
  1. **操作校验阶段**：基于 bscp 库内状态判定"无需停止/强制停止"时记录 829。
  2. **对比 GSE 状态阶段**：GSE 查询结果显示进程已停止、停止/强制停止操作无需执行时记录 829。
  3. **实际操作阶段**：GSE 操作返回错误码 829（`no need to stop ...`）时记录 829
     （现有逻辑已将 GSE 返回错误码原样记入，确认覆盖该码即可）。
- **输出**：子任务失败，且携带错误码 `829`，可被查询侧 `ignore_error_codes` 忽略。
- **边界条件**：仅 stop / kill 操作命中"已停止/无需停止"时记录 829。
- **异常处理**：其他失败原因不记录 829，保持真实失败。

## 业务规则

- **规则 R-001**：错误码 `828` 表示"重复启动"（进程已在运行），`829` 表示"重复停止/强制停止"
  （进程未在运行、无需停止）。停止（stop）与强制停止（kill）共用 829，判定逻辑一致。
- **规则 R-002**：命中重复启动/重复停止时，单个子任务仍标记为**失败**并保留错误信息供排查；
  仅在批次整体状态重判（`ignore_error_codes` 生效）时按成功计入。该行为与现有 828 一致。
- **规则 R-003**：是否忽略由调用方（SOPS 插件）通过查询接口 `GetTaskBatchDetail` 的
  `ignore_error_codes` 传入 `828`/`829` 决定；本需求不新增独立的接口参数。

## 边界范围

### 本期包含

- 校验阶段（ValidateOperate）对"重复启动"记录 828。
- 校验阶段、对比 GSE 阶段、实际操作阶段三处对"重复停止/强制停止"记录 829（stop 与 kill 一致）。
- 复用现有 `ignore_error_codes` 批次状态重判机制。

### 本期不包含

- register/unregister（重复托管/重复取消托管）的忽略处理。
- 新增独立的接口布尔参数（如 `ignore_duplicate`）。
- 查询侧 `ignore_error_codes` / 批次状态重判逻辑本身的改动（已具备能力，无需变更）。

## 验收标准

### 功能验收

- [ ] **AC-001**：Given 某进程在 bscp 库内状态为"运行中"，When 通过批量操作对其执行
  启动并在校验阶段被判定"无需启动"，Then 该子任务失败且错误码为 `828`。
- [ ] **AC-002**：Given 某进程已停止，When 在校验阶段对其执行停止（stop）或强制停止（kill）
  并被判定"无需停止"，Then 该子任务失败且错误码为 `829`。
- [ ] **AC-003**：Given 某进程在 GSE 侧已停止，When 在对比 GSE 状态阶段对其执行停止（stop）
  或强制停止（kill）被判定无需执行，Then 该子任务失败且错误码为 `829`。
- [ ] **AC-004**：Given 某进程已停止，When 实际操作阶段 GSE 返回错误码 829，
  Then 该子任务失败且错误码为 `829`。
- [ ] **AC-005**：Given 一个批次中同时存在"重复启动(828)/重复停止(829)"失败子任务和
  正常成功子任务，When 查询批次详情并传入 `ignore_error_codes=[828,829]`，
  Then 命中的失败子任务在批次整体状态重判时按成功计入，批次状态据此重新判定。
