## 需求背景

### 业务背景

bscp 进程管理通过 GSE 对业务进程进行托管（register/unregister），并在进程操作（启动/停止/重启/重载等）时下发到 GSE 执行。但 bscp DB 中记录的进程托管配置与 GSE 侧实际托管配置之间可能出现"漂移"：

- 用户在 GSE/外部渠道直接改动了托管配置，导致 bscp DB 与 GSE 不一致；
- 托管/取消托管过程中断或异常，导致状态与配置残留不一致；
- 存在非法托管项（GSE 上托管了 bscp 不认识的进程）。

当托管信息已经不一致时，若仍直接执行进程操作，可能作用在错误的配置上，造成误操作或操作失败，排障成本高。

参照系统 gsekit 已提供「检查进程配置托管信息」能力（`ProcessCheckManager`），可对整个业务的进程做托管配置比对并输出异常清单。bscp 希望具备类似能力，主动发现并记录托管异常，并在异常存在时阻断风险操作。

- **目标用户**：进程管理的运维/使用方，以及 bscp 平台自身的稳定性保障。
- **使用场景**：日常后台巡检发现托管配置不一致；用户发起进程操作前的安全校验。
- **不做的影响**：托管配置漂移无法被主动发现，错误操作风险持续存在，问题定位依赖人工。

### 用户故事

- 作为 **平台运维**，我想要 **系统定时巡检全业务进程的 GSE 托管配置一致性**，以便于 **在托管配置发生漂移时第一时间发现并定位**。
- 作为 **平台运维**，我想要 **将检查发现的异常进程及原因、处理建议持久化记录并保留历史**，以便于 **追溯问题、按建议处置**。
- 作为 **进程操作发起方**，我想要 **当被操作的进程实例处于托管异常态时操作被拒绝并给出明确原因与处理建议**，以便于 **避免在错误配置上执行操作造成误操作**。

### 需求来源

- **需求渠道**：技术优化 / 稳定性保障
- **关联需求**：无

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | 定时任务按业务维度逐个扫描全部进程实例，比对 GSE 实际托管配置与 bscp 期望托管配置的一致性 | P0 | 平台后台 | 完整配置比对，对标 gsekit |
| F-002 | 将检查发现的异常进程写入独立的「托管异常记录」表，保留历史明细 | P0 | 平台后台 | 含异常类型、原因、处理建议 |
| F-003 | 下一轮检查通过的进程，其异常记录自动解除（恢复正常） | P1 | 平台后台 | 自动闭环 |
| F-004 | 进程操作时，若被操作的进程实例自身处于托管异常态，则拒绝该操作并返回原因与处理建议 | P0 | 进程操作发起方 | 拦截全部操作类型 |

### 详细功能描述

#### [F-001] 定时托管配置一致性检查

- **输入**：定时触发（周期性）。检查对象为按业务维度遍历到的全部进程实例。
- **处理逻辑**：
  1. 主从选举：仅 master 实例执行（对齐现有 crontab 任务模式）。
  2. 按业务维度逐个处理：取该业务下全部进程实例及其在 bscp DB 中的期望托管配置（`ProcessInfo`：工作路径、PID 文件、启动/停止/重启/重载/强制停止命令、启动用户等）。
  3. 获取 GSE 侧实际托管配置：复用现有"配置检查"机制（`internal/task/executor/config/config_check.go`），通过 GSE 异步脚本执行接口（`AsyncExtensionsExecuteScript` / `async_execute_script` + `get_execute_script_result`）向目标 agent 下发脚本读取 agent 的 `.proc` 文件（对标 gsekit `cat .proc`），从返回 `Screen` 解析得到实际托管配置。
  4. 逐项比对期望配置与实际配置，识别以下异常类别（对标 gsekit）：
     - 已托管但 GSE 无托管信息；
     - 未托管但 GSE 存在托管信息；
     - 存在非法托管项（GSE 上托管了 bscp 期望之外的项）；
     - 托管配置属性不一致（工作路径/PID 文件/各类命令等字段存在差异）；
     - 实际信息获取/解析失败、agent 异常等。
  5. 检查结论用于 F-002（记录异常）与 F-003（恢复正常）。
- **输出**：每个进程实例的检查结论（正常 / 异常 + 异常类别 + 差异/原因 + 处理建议）。
- **边界条件**：
  - 期望配置直接取自 bscp DB，本期不为渲染期望配置而调用 CMDB。
  - 单业务进程/主机规模较大时需考虑分批与限流（见非功能需求）。
- **异常处理**：
  - 单个业务/主机检查失败 → 记录该范围异常，不阻断其余业务的检查。
  - GSE 调用失败 → 记录为对应异常类别，按处理建议给出排查方向。

#### [F-002] 异常进程记录（独立表 + 历史明细）

- **输入**：F-001 的异常检查结论。
- **处理逻辑**：将异常进程写入独立的「托管异常记录」表，保留历史明细。每条记录字段对标 gsekit：
  - 异常类型 `error_type`（如：解析失败 / agent 异常 / 非法 valuekey / 配置不符 / 已托管无信息 / 未托管有信息 等）；
  - 异常描述 `error_msg`（含具体差异信息）；
  - 处理建议 `handling_suggestion`；
  - 关联定位信息（业务、主机、进程实例等）与检查时间。
- **输出**：可查询、可追溯的托管异常历史记录。
- **边界条件**：保留历史明细（非覆盖式），便于追溯。
- **异常处理**：写库失败按平台通用错误处理记录日志。

#### [F-003] 异常自动恢复

- **输入**：后续轮次 F-001 的检查结论。
- **处理逻辑**：当某进程实例在后续检查中恢复一致（检查通过），自动解除其异常态（不再被判定为异常托管，从而不再拦截操作）。
- **输出**：异常态被清除，进程恢复可操作。
- **边界条件**：以"最近一次检查结论"为准判断当前是否异常。
- **异常处理**：—

#### [F-004] 异常托管进程的操作拒绝

- **输入**：用户发起的进程操作请求（全部操作类型：启动/停止/重启/重载/托管/取消托管/强制停止/删除等）。
- **处理逻辑**：在操作校验阶段判断**被操作的进程实例自身**是否处于托管异常态；若是，则拒绝本次操作。
- **输出**：拒绝结果，返回明确错误信息，附异常原因与处理建议（对标 gsekit 的 `handling_suggestion`）。
- **边界条件**：
  - 判定范围仅限被操作的进程实例自身，不因同主机/同业务存在其它异常进程而拦截。
  - 本期**不提供**"强制忽略/跳过"通道。
- **异常处理**：—

## 非功能需求

### 性能需求

- **执行周期**：周期性执行，周期可配置；默认值参照现有 crontab 任务（待确认，建议默认 20 分钟，对齐 `sync_cmdb`）。
- **规模与限流**：单业务进程/主机规模量级待补充；对 GSE 调用按现有 `rateLimiter` 模式限流，必要时分批处理，避免对 GSE 形成压力。

### 安全需求

- **权限控制**：沿用 bscp 现有进程操作鉴权，不放宽。
- **数据保护**：异常记录不落入敏感个人信息。

### 可用性与稳定性

- **容错能力**：单业务/单主机检查失败不影响其它业务检查；后台任务异常不影响在线进程操作主流程（仅在操作校验处读取异常态）。
- **主从**：仅 master 节点执行定时检查，避免重复扫描。

### 兼容性

- **接口兼容**：新增「托管异常记录」表与操作校验逻辑，不改变现有进程操作对外协议的成功路径；仅在异常托管态下新增拒绝分支。

## 业务规则

### 业务逻辑规则

- **规则 R-001**：托管配置比对维度为"完整配置一致性"，覆盖工作路径、PID 文件、启动/停止/重启/重载/强制停止命令、启动用户等。
- **规则 R-002**：异常判定以最近一次定时检查结论为准；检查通过即自动解除异常态。
- **规则 R-003**：操作拒绝仅针对"被操作的进程实例自身处于托管异常态"的情形，覆盖全部操作类型。
- **规则 R-004**：本期不提供强制忽略/跳过异常的操作通道。

### 数据校验规则

- 异常记录必填：异常类型、异常描述、处理建议、关联定位信息、检查时间。

### 权限规则

- 沿用现有进程管理角色与权限，不新增放宽。

## 验收标准

### 功能验收

- [ ] **AC-001**：Given 某进程实例 bscp 期望托管配置与 GSE 实际托管配置存在差异 When 定时检查执行 Then 该进程实例被判定为托管异常，并在「托管异常记录」表中新增一条记录（含异常类型、差异原因、处理建议）。
- [ ] **AC-002**：Given 某进程实例当前处于托管异常态且后续 GSE 实际配置已与期望一致 When 下一轮定时检查执行 Then 该进程实例异常态被自动解除，可正常操作。
- [ ] **AC-003**：Given 某进程实例当前处于托管异常态 When 用户对该进程实例发起任意操作（启动/停止/重启/重载/托管/取消托管/强制停止/删除等）Then 操作被拒绝，返回明确错误信息并包含异常原因与处理建议。
- [ ] **AC-004**：Given 某进程实例正常（无托管异常）但其同主机/同业务存在其它异常托管进程 When 用户对该正常进程实例发起操作 Then 操作不被拦截，正常执行。
- [ ] **AC-005**：Given 托管异常记录已存在历史多条 When 查询某进程实例的异常历史 Then 能看到保留的历史明细（非覆盖式）。

### 性能验收

- [ ] **AC-P01**：定时检查按业务逐个扫描，单轮检查对 GSE 的调用受限流约束，不对 GSE 造成异常压力（具体阈值随规模补充后确定）。

### 安全验收

- [ ] **AC-S01**：操作拒绝逻辑不绕过现有进程操作鉴权；异常记录不含敏感个人信息。

## 边界范围

### 本期包含

- 定时按业务扫描全部进程实例的完整托管配置一致性检查（F-001）。
- 独立「托管异常记录」表与历史明细记录（F-002）。
- 异常自动恢复（F-003）。
- 被操作进程实例自身异常态时拒绝全部操作类型（F-004）。

### 本期不包含

- "强制忽略/跳过"异常继续操作的通道。
- 因同主机/同业务存在其它异常进程而拦截操作。
- 为渲染期望配置而调用 CMDB。
- 异常的前端展示/告警通知（如需另立需求，本文聚焦后台能力）。

## 约束条件

- **技术限制**：定时任务接入现有 `cmd/data-service/service/crontab/` 框架（主从选举 + ticker）；操作拒绝接入现有进程操作校验阶段。
- **数据来源限制**：期望配置取自 bscp DB，本期不调用 CMDB。
