## 需求背景

### 业务背景

BSCP 通过 `BuildProcessOperate` 构建 GSE `ProcessOperate` 对象，再下发到 GSE 完成进程的启动、停止、重启等托管操作。该构建逻辑参考自蓝鲸 gsekit（bk-process-config-manager）的进程操作实现。

当前实现存在两处与 gsekit 不一致的行为，导致进程在重启场景下出现异常（表现为进程重复启动 / 重启失败）：

1. **强制下发资源限制**：`BuildProcessOperate` 始终为每个进程下发 CPU 使用率上限（`DefaultCPULimit = 30.0`）和内存使用率上限（`DefaultMemLimit = 10.0`）。而 gsekit 原实现并不下发资源限制。该资源限制会在进程重启时触发 GSE 侧的存活/资源校验异常，导致重启失败，进而引发进程重复启动问题。
2. **缺少操作参数缺省值**：`StartCheckSecs`（启动后存活检查时间）与 `OpTimeout`（命令执行超时时间）直接取自用户配置，当用户未配置（值为 0）时不会套用 gsekit 的缺省值，导致 GSE 在执行启动/重启命令时按 0 秒检查或 0 秒超时，校验时机不合理，加剧重启失败。

gsekit 对应缺省值：

```python
# 参数缺省值
DEFAULT_START_CHECK_SECS = 5
DEFAULT_OP_TIMEOUT = 60
```

不修复会持续出现进程重启失败、进程重复启动，影响进程托管的可靠性。

### 用户故事

作为 BSCP 进程托管的使用者
我想要进程在重启时不被多余的资源限制和不合理的检查/超时参数影响
以便于进程能够稳定重启，不再出现重启失败导致的进程重复启动

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | 构建 GSE `ProcessOperate` 时不再下发 CPU/Mem 资源限制 | P0 | 系统 | 必须，与 gsekit 对齐 |
| F-002 | `StartCheckSecs` 未配置时套用缺省值 5 | P0 | 系统 | 必须 |
| F-003 | `OpTimeout` 未配置时套用缺省值 60 | P0 | 系统 | 必须 |

### 详细功能描述

#### [F-001] 移除资源限制下发

- **输入**：进程操作构建请求（任意 GSE 操作类型）
- **处理逻辑**：
  1. 构建 `ProcessOperate` 时，不再设置 CPU/Mem 资源使用率上限
  2. 与 gsekit 原实现完全对齐，彻底不下发资源限制字段
- **输出**：下发给 GSE 的 `ProcessOperate` 不携带 CPU/Mem 资源限制
- **边界条件**：
  - 适用于所有操作类型（启动、停止、重启、托管、重载等）
- **异常处理**：
  - 无新增异常分支，仅去除原有资源限制赋值

#### [F-002 / F-003] 操作参数缺省值

- **输入**：进程配置信息 `ProcessInfo`（含 `StartCheckSecs` 与 `Timeout`）
- **处理逻辑**：
  1. 当 `StartCheckSecs` 未配置（值为 0）时，套用缺省值 5；用户已配置（非 0）则尊重用户配置
  2. 当 `OpTimeout`（对应 `ProcessInfo.Timeout`）未配置（值为 0）时，套用缺省值 60；用户已配置（非 0）则尊重用户配置
- **输出**：下发给 GSE 的 `MonitorPolicy` 中 `StartCheckSecs` 与 `OpTimeout` 均为有效值
- **边界条件**：
  - 缺省值仅在「值为 0 / 未设置」时生效，不覆盖用户的显式配置
- **异常处理**：
  - 无新增异常分支

## 业务规则

### 业务逻辑规则

- **规则 R-001**：进程操作下发不包含 CPU/Mem 资源使用率上限，行为与 gsekit 一致。
- **规则 R-002**：`StartCheckSecs` 缺省值为 5，仅当入参值为 0（未配置）时生效。
- **规则 R-003**：`OpTimeout` 缺省值为 60，仅当入参值为 0（未配置）时生效。
- **规则 R-004**：用户显式配置的 `StartCheckSecs` / `OpTimeout`（非 0）必须被尊重，不被缺省值覆盖。

## 验收标准

### 功能验收

- [ ] **AC-001**：Given 任意进程操作构建请求 When 构建 GSE `ProcessOperate` Then 输出对象中不携带 CPU/Mem 资源限制
- [ ] **AC-002**：Given `StartCheckSecs` 入参为 0（未配置） When 构建 `ProcessOperate` Then `MonitorPolicy.StartCheckSecs` 为 5
- [ ] **AC-003**：Given `OpTimeout`（`ProcessInfo.Timeout`）入参为 0（未配置） When 构建 `ProcessOperate` Then `MonitorPolicy.OpTimeout` 为 60
- [ ] **AC-004**：Given `StartCheckSecs` 入参为非 0 值 N When 构建 `ProcessOperate` Then `MonitorPolicy.StartCheckSecs` 为 N（不被缺省值覆盖）
- [ ] **AC-005**：Given `OpTimeout` 入参为非 0 值 M When 构建 `ProcessOperate` Then `MonitorPolicy.OpTimeout` 为 M（不被缺省值覆盖）
- [ ] **AC-006**：Given 进程曾因资源限制重启失败 When 应用本修复后执行重启 Then 进程能正常重启，不再出现重复启动

## 边界范围

### 本期包含

- 移除 GSE 进程操作下发中的 CPU/Mem 资源限制
- 为 `StartCheckSecs` 与 `OpTimeout` 提供缺省值（5 / 60），仅在未配置时生效

### 本期不包含

- 资源限制的可配置化能力（本期彻底移除，不做开关或后续配置入口）
- `StopCheckSecs` 等其它监控策略参数的缺省值调整
- GSE 进程操作其它字段的行为变更

## 约束条件

- **兼容性约束**：行为需与 gsekit（bk-process-config-manager V1.0.X）的进程操作实现保持一致。
- **技术规范约束**：Go 代码需符合仓库 `.golangci.yml`，修改后运行 `gofmt`，并补充/更新单包测试。
