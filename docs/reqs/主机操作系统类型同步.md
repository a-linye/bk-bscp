## 需求背景

### 业务背景

BSCP 通过定时任务从蓝鲸 CMDB（CC）同步业务下的主机与进程拓扑数据。当前 `process` 表、`Host` 结构、`process.proto` 中均已存在 `os_type`（操作系统类型）字段，进程同步比较逻辑（`BuildProcessChanges`）也已纳入 `osTypeChanged` 判断。

但 CMDB 数据同步的新版接口在拉取主机数据时未正确同步主机的操作系统类型，导致同步入库后的主机及其关联进程的 `os_type` 字段为空，进程详情中无法展示正确的操作系统类型，影响数据完整性与后续基于操作系统类型的运营/展示能力。

### 用户故事

作为 BSCP 平台运维/使用者
我想要在主机与进程数据从 CMDB 同步入库时，能携带并落地正确的操作系统类型（`os_type`）
以便于在进程详情等场景准确看到主机所属的操作系统类型（Linux / Windows / AIX）

### 需求来源

- **需求渠道**：技术优化 / 数据完整性修复
- **关联需求**：无
- **参考资料**：CMDB（CC）`list_hosts` 接口（`%s/api/v3/hosts/app/%d/list_hosts`）请求/响应示例见"原需求描述"

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | CMDB 主机数据同步时获取主机 `bk_os_type` | P0 | 同步任务 | 必须 |
| F-002 | 将 `bk_os_type` 映射为业务语义后落库到主机维度 | P0 | 同步任务 | 必须 |
| F-003 | 主机维度获取后，关联写入进程 `process.os_type` | P0 | 同步任务 | 必须 |

### 详细功能描述

#### [F-001] 同步主机操作系统类型

- **输入**：定时同步任务触发，按业务/模块从 CMDB 拉取主机列表
- **处理逻辑**：
  1. 通过 CMDB `list_hosts` 接口（`%s/api/v3/hosts/app/%d/list_hosts`）拉取主机，请求 `fields` 中包含 `bk_os_type`
  2. 主机可通过云区域（`bk_cloud_id`）与内网 IP（`bk_host_innerip`）唯一确定
  3. 读取响应中每台主机的 `bk_os_type`
- **输出**：每台主机带有 `bk_os_type` 原始值
- **边界条件**：
  - `bk_os_type` 可能为 `null` 或空
- **异常处理**：
  - 接口异常时沿用同步任务现有失败处理逻辑（本需求不改变同步任务的整体重试/容错机制）

#### [F-002] 操作系统类型映射与落库（主机维度）

- **输入**：主机的 `bk_os_type` 原始值
- **处理逻辑**：
  1. 按映射规则将 CC 数字编码转换为业务语义字符串（见业务规则 R-001）
  2. 写入主机维度 `os_type` 字段
- **输出**：主机 `os_type` 为 `linux` / `win` / `aix` 之一，或空字符串
- **边界条件**：
  - 原始值为 `null`/空：落库为空字符串，且不覆盖已有的非空值（仅在取到非空值时更新）
- **异常处理**：
  - 映射表未覆盖的编码：落空字符串（不阻断同步），具体取舍见未解决问题 Q-001

#### [F-003] 进程关联写入操作系统类型

- **输入**：主机维度已获取的 `os_type`
- **处理逻辑**：
  1. 同步进程数据时，按主机关联关系将主机 `os_type` 写入 `process.os_type`
  2. 复用现有进程变更比较逻辑（`osTypeChanged`：仅当新值非空且与旧值不同才更新）
- **输出**：进程 `os_type` 与其所在主机一致
- **边界条件**：
  - 主机 `os_type` 为空时，不覆盖进程已有非空 `os_type`

## 业务规则

### 业务逻辑规则

- **规则 R-001（os_type 映射）**：CC 数字编码 → 业务语义字符串
  - `1` → `linux`
  - `2` → `win`
  - `3` → `aix`
  - 其余编码 / `null` / 空：落空字符串（详见 Q-001）
- **规则 R-002（空值不覆盖）**：当本次同步取到的 `os_type` 为空时，不覆盖已有的非空值；仅在取到非空映射值时更新。
- **规则 R-003（存量处理）**：本需求仅对后续同步生效，存量数据由下次定时同步自然补齐，不做专门的全量回刷。

### 数据校验规则

- **取值范围**：`os_type` ∈ {`linux`, `win`, `aix`, ""}

## 验收标准

### 功能验收

- [ ] **AC-001**：Given CMDB 中某主机 `bk_os_type=1` When 触发 CMDB 数据同步 Then 该主机入库后 `os_type=linux`，其关联进程 `process.os_type=linux`
- [ ] **AC-002**：Given CMDB 中某主机 `bk_os_type=2` When 触发同步 Then 主机及关联进程 `os_type=win`
- [ ] **AC-003**：Given CMDB 中某主机 `bk_os_type=3` When 触发同步 Then 主机及关联进程 `os_type=aix`
- [ ] **AC-004**：Given CMDB 返回某主机 `bk_os_type=null` 且该主机已有非空 `os_type` When 触发同步 Then 不覆盖原有 `os_type`（保持非空值）
- [ ] **AC-005**：Given CMDB 返回某主机 `bk_os_type=null` 且该主机此前无 `os_type` When 触发同步 Then `os_type` 为空字符串，同步不报错
- [ ] **AC-006**：Given 存量主机 `os_type` 为空 When 下一次定时同步取到非空 `bk_os_type` Then 该主机 `os_type` 被自动补齐（无需人工回刷）

### 展示验收

- [ ] **AC-D01**：在进程详情中可看到与其所在主机一致的、正确映射后的操作系统类型

## 边界范围

### 本期包含

- CMDB 同步链路获取并映射主机 `bk_os_type`
- 主机维度落库 `os_type`，并关联写入进程 `process.os_type`
- 支持 `linux` / `win` / `aix` 三类映射
- 空值不覆盖已有非空值

### 本期不包含

- 存量历史数据的专门全量回刷（依赖后续定时同步自然补齐）
- 基于 `os_type` 的下游配置下发差异化逻辑（本期仅做数据补全/展示，不改变下发行为）
- `process.proto` / 对外 API 字段结构变更（字段已存在）
