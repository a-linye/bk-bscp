## 缺陷现象

插件通过蓝鲸 API 网关调用 bscp 接口发布版本时，接口返回 `record not found`，发布流程无法完成。

## 复现条件

- 部署环境：**非多租户的上云环境**（`featureFlags.enableTenantMode = false`）。
- 调用方式：插件经蓝鲸 API 网关调用 bscp 发布相关接口。
- 触发前提：目标应用开启了 ITSM 审批（`app.Spec.IsApprove = true`）且 ITSM 走 V4（`ITSM.EnableV4 = true`）。

> 待确认：当前缺乏精确的请求样例与日志片段（含 record not found 命中的具体 key），如有可补充至「未解决问题」并用于回归比对。

## 根因分析

问题本质是 **ITSM stateID 配置在「写入时」与「读取时」使用的租户标识不一致**：

1. **写入配置时**（`scripts/migrations/itsm/v4.go`、`cmd/data-service/service/crontab/itsm_v4_tenant_template.go`）
   以 `kit.TenantID` 拼接 key 前缀。非多租户上云环境注册时 `TenantID` 为空，key 落库为无前缀形式，例如：
   - `create_or_sign_approve_itsm_state_id`
   - `create_count_sign_approve_itsm_state_id`

2. **读取配置时**（`cmd/data-service/service/publish.go:260/1002/1286`、`crontab/sync_ticket_status.go:153`）
   蓝鲸 API 网关在非多租户上云环境会默认向请求注入 header `X-Bk-Tenant-Id: default`，`pkg/kit/kit.go` 将其解析为 `grpcKit.TenantID = "default"`。`internal/components/itsm/service.go` 的 `BuildStateIDKey` 因 `tenantID != ""` 而拼出带前缀的 key：
   - `default-create_or_sign_approve_itsm_state_id`

3. configs 表中不存在 `default-` 前缀的 key，`s.dao.Config().GetConfig` 返回 `record not found`，发布失败。

核心矛盾：写入侧把「非多租户」视为空租户（无前缀），读取侧却把网关注入的 `default` 当作真实租户（带前缀），两侧口径不一致。

## 期望行为

- 在非多租户上云环境下，插件经网关发布版本时应能正常读取 ITSM stateID 配置并完成发布/审批流程，行为与 Web 端直接发布一致。
- **口径统一（本次确认结论）**：将网关注入的 `default` 归一化为「无租户」，即读取配置时不为 `default` 追加前缀，与现有写入侧保持一致。

## 影响范围

- **本次聚焦**：非多租户（`default`）上云环境下的发布失败问题。
- **回归覆盖**：归一化口径需覆盖全部 4 处 `BuildStateIDKey` 调用入口，保证不同流程下行为一致：

| 入口 | 文件位置 | 场景 |
|------|---------|------|
| 1 | `cmd/data-service/service/publish.go:260` | 发布审批状态校验 |

| 3 | `cmd/data-service/service/publish.go:1286` | 策略发布 |
| 4 | `cmd/data-service/service/crontab/sync_ticket_status.go:153` | 定时同步 ITSM ticket 状态 |

## 业务规则

- **规则 R-001**：租户标识为 `default`（蓝鲸默认租户）时，等价于「无租户」，在以 TenantID 拼接 ITSM 配置 key 的逻辑中不追加前缀。
- **规则 R-002**：多租户场景（真实租户 ID，非 `default`/空）行为保持不变，仍按 `<tenantID>-` 前缀读取配置。
