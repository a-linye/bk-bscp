## 需求背景

### 业务背景

父需求要求对全业务进程的 GSE 托管配置做一致性检查，并将检查发现的异常进程持久化记录、保留历史明细，供追溯与按建议处置；同时进程操作侧需要据此判断进程实例当前是否处于托管异常态。

本子需求是上述能力的**数据基础**：定义并实现「托管异常记录」的数据模型与读写存储，使检查侧（写入/恢复）与操作侧（读取异常态）有统一、可追溯的数据载体。聚焦数据层，不含检查逻辑与操作拦截逻辑。

### 用户故事

作为 **平台运维**
我想要 **将进程托管检查发现的异常以独立记录持久化并保留历史明细**
以便于 **后续追溯每次异常的类型、原因与处理建议，并支撑操作侧的异常态判断**

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | 新增独立「托管异常记录」表及其数据模型 | P0 | 平台后台 | 字段对标 gsekit |
| F-002 | 提供异常记录的写入、查询、状态更新（恢复）DAO 能力 | P0 | 平台后台 | 供检查侧与操作侧复用 |

### 详细功能描述

#### [F-001] 托管异常记录数据模型

- **输入**：无（建表/模型定义）。
- **处理逻辑**：
  1. 新增独立的「托管异常记录」表，字段对标 gsekit 并满足定位与追溯需求：
     - 异常类型 `error_type`（解析失败 / agent 异常 / 非法 valuekey / 配置不符 / 已托管无信息 / 未托管有信息 等枚举）；
     - 异常描述 `error_msg`（含具体差异信息）；
     - 处理建议 `handling_suggestion`；
     - 关联定位信息：租户、业务、主机、进程实例（process_instance）等；
     - 记录状态（异常 / 已恢复）与检查时间、更新时间。
  2. 历史明细保留：以追加方式保存（非覆盖），可按进程实例追溯历史多条记录。
- **输出**：可被检查侧写入、被操作侧读取的持久化表结构。
- **边界条件**：保留历史明细，不做物理覆盖；表结构需兼容大表查询（按业务/进程实例的索引）。
- **异常处理**：建表/迁移失败按平台通用迁移流程处理。

#### [F-002] 异常记录读写 DAO 能力

- **输入**：检查结论（写入/状态更新）、查询条件（按进程实例/业务）。
- **处理逻辑**：提供以下能力：
  1. 写入一条新的异常记录；
  2. 查询某进程实例/业务的异常记录（含"当前是否处于异常态"的判定查询）；
  3. 将异常记录状态更新为"已恢复"（供下一轮检查通过时调用）。
- **输出**：DAO 接口返回结果。
- **边界条件**：以"最近一次记录状态"判断当前是否异常。
- **异常处理**：写库失败记录日志并返回错误，由调用方决定是否阻断。

## 非功能需求

### 性能需求
- **查询性能**：按业务 / 进程实例查询需走索引，避免全表扫描。

### 安全需求
- **数据保护**：异常记录不落入敏感个人信息。

## 验收标准

### 功能验收

- [ ] **AC-001**：Given 检查侧调用写入接口传入一条异常（含类型/描述/建议/定位信息）When 写入执行 Then 「托管异常记录」表新增一条记录，字段完整且可查询。
- [ ] **AC-002**：Given 同一进程实例先后写入多条异常记录 When 查询该进程实例异常历史 Then 返回全部历史记录（非覆盖式）。
- [ ] **AC-003**：Given 某进程实例存在状态为"异常"的记录 When 调用状态更新接口置为"已恢复" Then 该记录状态变为"已恢复"，且"当前是否异常"判定查询返回否。
- [ ] **AC-004**：Given 某进程实例无异常记录或最近记录为"已恢复" When 调用"当前是否异常"判定查询 Then 返回否。

## 边界范围
