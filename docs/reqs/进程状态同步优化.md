## 需求背景

### 业务背景

BSCP 通过「按业务全量同步 GSE 进程状态」能力，将业务下所有进程实例的运行状态（运行/停止/启动中/停止中）与托管状态从 GSE 同步回本地数据库。该能力当前有两个入口：

- **接口触发**：`POST /api/v1/config/biz_id/{biz_id}/sync/cmdb_gse_status`，内部创建异步任务（先 `SyncCMDB` 再 `SyncGseStatus`），接口立即返回 `task_id`。
- **定时任务触发**：`syncCmdbGse` 定时任务周期性执行同一套同步流程。

实际同步逻辑位于 `internal/processor/gse/sync_gse.go` 的 `SyncSingleBiz`。与蓝鲸 gsekit（bk-process-config-manager）的 `sync_biz_process_status` 相比，BSCP 的同步**完成耗时明显更长**，在进程/实例数量较多的业务上尤为突出。根因在于两者对 GSE 任务的批处理粒度与并发模型差异较大：

| 维度 | BSCP 现状（`SyncSingleBiz`） | gsekit（`sync_biz_process_status`） |
|------|------------------------------|-------------------------------------|
| GSE 任务粒度 | **每个进程**下发一个 `OperateProcMulti` 任务 | 将全业务进程实例按 **1000 实例/批** 合并为少数任务 |
| 任务数量 | 与进程数同量级（可达数百） | 实例数 / 1000，通常仅个位数 |
| 并发模型 | errgroup 并发上限 **5** | 线程池并发，任务数本就很少 |
| 轮询参数 | 间隔 3s、最长 15s | 间隔 1.5s |
| 附带开销 | 每进程额外一次 `GetByProcessIDs` DB 查询 | 一次性批量取实例 |

任务数量与并发模型相差一两个数量级，导致 BSCP 同步整体耗时随进程数近似线性放大且常数很大。本需求即针对该「按业务全量同步」链路进行性能优化，使其完成速度对齐 gsekit 量级，同时保证同步结果正确、不破坏既有进程托管行为。

### 详细功能描述

#### [F-001] 按业务全量同步性能优化

- **输入**：业务 ID（及租户）触发的全量同步请求（接口或定时任务）
- **处理逻辑**（目标行为，不锁定具体实现方案）：
  1. 汇总业务下全部进程实例，减少 GSE 任务数量（不再「每进程一个任务」）
  2. 提升并发能力，缩短整体等待时间
  3. 优化轮询节奏，避免不必要的等待
  4. 消除每进程额外的冗余 DB 查询
- **输出**：进程/实例状态被正确批量更新，整体同步完成耗时显著下降
- **边界条件**：
  - 实例规模较大时（数千实例）仍能稳定完成，不因批量过大导致超时/内存异常
  - 对 GSE 的并发压力对齐 gsekit（按批合并、任务数少），无需额外限流上限
- **异常处理**：
  - 单个进程/批次失败不阻断整体同步（沿用现状的容错策略）
  - 超时/取消时返回明确错误，不写入不完整的状态结果

#### [F-002] 同步结果正确性

- **处理逻辑**：状态解析规则（`ErrCodeSuccess` / `ErrCodeStopping` / `115 进行中` 等）与现状保持一致
- **输出**：`status`、`managed_status`、`process_state_synced_at` / `status_updated_at` 等字段语义不变

## 非功能需求

### 性能需求

- **目标**：按业务全量同步整体完成耗时对齐 gsekit 量级（相同业务规模下显著优于现状）。
- **量化指标**（第 2 轮澄清确认）：对齐 gsekit 量级即可，本期不设硬性秒级指标；以「相同业务规模下相比现状显著下降、与 gsekit 同类能力同量级」作为达标判据。

### 可用性与稳定性

- 大规模业务下同步不超时失败、不 OOM。
- 对 GSE 的请求压力可控，避免触发 GSE 侧限流。

### 兼容性

- 接口契约不变（第 2 轮澄清确认：维持异步、返回 `task_id`，不改变返回方式）。
- 同步结果数据结构与字段语义保持向后兼容。

## 业务规则

- **规则 R-001**：本期优化范围限定为「按业务全量同步」(`SyncSingleBiz` / `SyncGseStatus` step)。
- **规则 R-002**：优化不得改变进程/实例状态的判定结果，仅改变同步的执行方式与速度。
- **规则 R-003**：不得破坏缺陷「进程重复启动修复」中关于资源限制、`StartCheckSecs`、`OpTimeout` 的行为。

## 验收标准

### 功能验收

- [ ] **AC-001**：Given 一个包含较多进程/实例的业务 When 触发按业务全量同步 Then 同步整体完成耗时相比现状显著下降（对齐 gsekit 量级）
- [ ] **AC-002**：Given 同步完成 When 对比优化前后的同步结果 Then 进程/实例的 `status`、`managed_status`、同步时间字段一致且正确
- [ ] **AC-003**：Given 某进程或某批次同步失败 When 执行全量同步 Then 不阻断其它进程的同步，整体仍可完成
- [ ] **AC-004**：Given 已应用「进程重复启动修复」 When 执行本优化后的同步 Then 进程托管行为不受影响，不出现重复启动/重启失败
- [ ] **AC-005**：Given 数千实例规模的业务 When 触发同步 Then 不超时、不 OOM，正常完成

### 性能验收

- [ ] **AC-P01**：Given 相同业务规模 When 对比优化前后的按业务全量同步耗时 Then 优化后整体完成耗时相比现状显著下降，且与 gsekit `sync_biz_process_status` 处于同一量级（本期不设硬性秒级阈值）
- [ ] **AC-P02**：Given 业务下进程数较多 When 触发同步 Then GSE 任务数量从「与进程数同量级」下降为「实例数 / 批大小」量级（不再每进程一个任务）

## 边界范围

### 本期包含

- 优化「按业务全量同步」(`SyncSingleBiz`) 链路的性能（批处理粒度、并发、轮询、冗余 DB 查询）
